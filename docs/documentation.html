1Overview

vibematch is a community application for Bilkent University students. A user signs up with their Bilkent email, verifies it with a code, selects their interests, and takes a short 16-question personality test. The application then recommends communities (clubs, societies, study circles) that match the user's interests and personality. Each community has a forum with threaded discussions, and users can add friends and exchange direct messages.

The application is written in Java. It is built as a client/server system: a server program holds a single shared database and pushes real-time updates, and a desktop client (Java Swing) connects to it over a network socket. This document explains what was built, in what order, and the reasoning behind each decision.

2Design approach — dependency-first

The whole system was built bottom-up, following one rule: a component is never built before the components it depends on. This is the same logic as constructing a building — the foundation is poured before the walls are raised, and the plumbing is installed before the painting.

The code is organised into layers, and each layer depends only on the layer below it. This keeps the system maintainable: a change in one layer does not break the layers above it. From bottom (built first) to top:

Layer 1  model        plain data classes            (the foundation)
Layer 2  data         database + DAO classes        (all SQL lives here)
Layer 3  service      business rules
Layer 4  server       socket server + handlers      (network / concurrency)
Layer 5  net          client-side connection (Api)
Layer 6  ui / screens what the user sees

Figure 1. Layered architecture. A request travels up from the screen; data returns back down. See uml.html for the full class diagram.

The project history on GitHub is roughly forty small commits, each adding a single component in this order. The commit history is itself evidence that the system was built incrementally rather than all at once.

3Build order

  1. Model (data classes). User, Community, Post, Comment, Message. Built first because every other layer uses them; you cannot design storage without knowing what is being stored.
  2. Database and DAOs. An SQLite database and one data-access class per table. All SQL is confined to these classes, so the storage engine could be replaced without touching anything above.
  3. Services (rules). AuthService (Bilkent-email rule), MatchService (scoring), MbtiService. Built once data access existed, so rules could be layered on top of it.
  4. Identity — the first real feature. Register, login, email verification. Built first among features because, without a user, nothing has an owner — no message, no membership, no match. Identity is the backbone.
  5. Onboarding, communities, forum, messages. In this order, because matching needs the vibe, the forum needs communities, and messaging needs users. Each feature depends on the previous one.
  6. Client/server conversion. Only after the logic worked was the data moved behind a socket server. Correctness is verified first, then the working system is placed on the network.
  7. Speed, friends, notifications. Added last, because the core already functioned; these are refinements.

5Email verification

Only Bilkent students may register, and the email is confirmed before the account can be used. The sequence is:

  1. The user enters a name, a Bilkent email, and a password.
  2. AuthService.register() checks the email domain and that the username and email are not already taken, then saves the account as unverified.
  3. The server generates a random 6-digit code and keeps it in memory.
  4. EmailSender sends the code by SMTP (JavaMail). If SMTP is not configured, the code is shown in a dialog so the application still works.
  5. The user enters the code; the server compares it and marks the account verified. Login is refused until the account is verified.

Why the dialog fallback?

Sending real email requires a private password, which is never committed to the repository. The fallback lets the application run and be evaluated on a machine with no email credentials, without exposing any secret.

How is the password stored?

Never in plain text. Each user has a random salt, and the stored value is a SHA-256 hash of the salt combined with the password (PasswordUtil). The server never sends the hash to a client.

6Sockets and real-time communication

What a socket is. A socket is a persistent two-way connection between two programs over a network. Once open, either side may send data at any time — comparable to a telephone call that stays connected, rather than sending a single letter and waiting for a reply.

Why it is needed. A community application has many users on different computers who must see the same data. This requires (a) a single shared store for the data, and (b) a way for the server to notify a user that something has happened without the user repeatedly asking. A persistent socket to a central server provides both. It also avoids paying for a hosted service: the server is itself the shared database and the real-time channel.

How it works here. The server was written with plain Java sockets (java.net.ServerSocket), with no external networking library. Messages are single lines of JSON.

// server accept loop (ChatServer)
while (true) {
    Socket s = serverSocket.accept();          // wait for a client
    new Thread(new ClientHandler(s)).start();  // one thread per client
}

// request / reply, matched by id
client -> {"id":4,"action":"login","data":{...}}
server -> {"type":"reply","id":4,"ok":true,"data":{...}}

// real-time push: the server sends without being asked
// when a direct message is sent to you:
ChatServer.pushTo(you, {"type":"push","event":"newMessage",...})
// it arrives on your socket immediately; no polling, no reloading

The server keeps a map of username → socket for connected users, so it knows where to deliver a push. Real-time messaging and forum updates use this mechanism.

Why plain sockets rather than a framework?

They are simpler to explain, have no dependencies, and represent classic socket programming. The same server can later serve a second client (for example an iOS application) without any change to the server itself.

7Friends and privacy

A user cannot message an arbitrary person. Messaging is restricted to friends, following the model people already know from other applications:

  1. A friend request is sent and stored with status pending.
  2. The recipient receives a real-time push and a notification.
  3. When the recipient accepts, the status becomes accepted.
  4. Only then does MessageHandler.send() permit a message between the two users. A message to a non-friend is refused by the server.

The rule is enforced on the server, not in the screen, because the screen can be bypassed but the server cannot. The check lives in one place, so it holds for the current desktop client and any future client.

8Notifications

Friend requests, accepted requests, new messages, and replies each create a stored notification and push a live update. Stored notifications remain available when the user logs in later; the live push updates the notification bell's unread count immediately for a user who is online.

9Summary

vibematch was built in the direction of its dependencies: model, then storage, then rules, then identity, then content, then interaction, then the network, and finally the refinements. No component was built before the component it depends on. The layered structure keeps each part independent, the socket server provides shared data and real-time updates, and the friendship rule provides privacy. The class diagram is available at uml.html.

Building vibematch,
brick by brick.

A Bilkent student community app. This page explains what we built, in what order, and why — the design decisions behind every part.

01 · What it is

A place to find your people

You sign up with your Bilkent email, verify it with a code, pick your interests, take a short personality test, and vibematch recommends communities (clubs, societies, study circles) that fit you. Each community has a Reddit-style forum, and you can add friends and message them 1-on-1.

✉️

Verified sign-up

Only @bilkent emails, confirmed by a code.

🧭

Vibe matching

Interests + a 16-question test → a match score.

💬

Forum + DMs

Community discussions and friend messaging.

🔔

Real-time

Messages and notifications arrive instantly.

02 · Our approach

We built it dependency-first

The single rule that shaped everything: you cannot build a thing before the thing it needs. Like a building — you pour the foundation before you raise the walls, plumb before you paint.

So we always worked bottom to top: first the data, then how it's stored, then the rules, then the screens, and only at the very end the networking and the nice-to-haves. Every layer only knows the one beneath it, so changing one part never breaks the others.

03 · Build order

First this, then that — and why

Model — the data classes

User, Community, Post, Comment, Message. Plain objects with fields.

Why first? Everything uses them. You can't design storage before you know what you're storing.

Database + DAOs

An SQLite database, and one DAO class per table holding all the SQL.

Why here? Model is ready, so each model gets a table. SQL stays in one place so it's swappable.

Services — the rules

AuthService (Bilkent-email rule), MatchService (scoring), MbtiService.

Why here? Now that we can read/write data, we can put rules on top of it.

Identity — the first real feature

Register, login, email verification.

Why first feature? Without a user, nothing has an owner — whose message, whose membership? Identity is the backbone.

Onboarding → communities → forum → messages

Interests + vibe test, then matched communities, then discussions, then DMs.

Why this order? Matching needs the vibe; the forum needs communities; messages need users. Each leans on the previous.

Turn it into a client/server app

Only after the logic worked did we move the data behind a socket server.

Why so late? You make sure the logic is correct first, then put it on the network. Moving a broken thing onto a network is twice the trouble.

Speed, friends, notifications — last

Background loading, friend requests, the notification bell.

Why last? The core already works; these are the polish and the paint.

04 · A closer look

How email verification works

We only let Bilkent students in, and we prove the email is really theirs before the account becomes usable.

// the flow, step by step 1. user types name + @ug.bilkent.edu.tr + password 2. AuthService.register() checks: right domain? username free? → save (unverified) 3. server makes a random 6-digit code, keeps it in memory 4. EmailSender mails the code over SMTP (JavaMail) // if SMTP isn't set up, the code shows in a popup so it still works 5. user types the code → server compares → marks the account verified 6. login is refused until verified == true
Q: Why keep a fallback popup?
So the app runs and can be graded even on a machine with no email credentials set up. Real emails need a private password we never commit to GitHub, so the fallback keeps it working without leaking secrets.
Q: Where is the password stored?
Never in plain text. We store a salted SHA-256 hash (PasswordUtil). Each user gets a random salt; we hash salt + password. The server never even sends the hash to a client.
05 · The important one

Sockets: what, why, how

What is a socket?

A socket is a live, two-way pipe between two programs over the network. Once it's open, either side can send bytes down it at any time — like a phone call that stays connected, versus sending one letter and waiting for a reply.

Why did we need one?

A community app has many people on different computers who must see the same thing. That needs two things a single desktop program can't do: (1) one shared place for all the data, and (2) a way for the server to say "something new happened" without the user constantly asking. Both are solved by keeping an open socket to a central server.

It also means we don't pay for a service like Supabase — our server is our shared database + real-time, and one server can host many apps.

How ours works

We wrote the server ourselves with plain Java sockets (java.net.ServerSocket) — no external networking library. Messages on the wire are single lines of JSON.

// server side (ChatServer) while (true) { Socket s = serverSocket.accept(); // wait for a client new Thread(new ClientHandler(s)).start(); // one thread each → many at once } // request / reply (client asks, server answers, matched by id) client → {"id":4,"action":"login","data":{...}} server → {"type":"reply","id":4,"ok":true,"data":{...}} // the real-time part: server pushes on its own someone sends you a DM → ChatServer.pushTo(you, {"type":"push","event":"newMessage",...}) // it goes straight down your socket — you didn't ask, no refreshing
🧵

Thread per client

Each connection runs on its own thread, so many people are online at once.

🗺️

Online map

The server tracks username → socket so it knows where to push.

⚡

Push, not poll

New messages/notifications are pushed instantly — we never re-query the DB in a loop.

Q: Why plain sockets and not a big framework?
It's clearer to explain and defend, has zero dependencies, and it's classic socket programming. The same design also lets a future iOS app connect to the same server — the server never changes.
06 · Friends & privacy

Not everyone can message everyone

Random people shouldn't be able to DM you. So messaging is gated by friendship, just like the apps people already know.

1. you send a friend request (FriendDao stores it as "pending") 2. the other person gets a real-time push + a notification 3. they accept → status becomes "accepted" 4. only now does MessageHandler.send() allow a DM between you // a message to a non-friend is refused by the server

Notifications work the same way: friend requests, accepted requests, new messages and replies each create a stored notification and push a live update, so the bell's unread count changes on its own.

Q: Why enforce it on the server, not the screen?
Because the screen can be bypassed, the server can't. The rule lives in one place (MessageHandler.send) so it always holds, for the desktop app today and any client tomorrow.

The one sentence to defend it

“We built it in the direction of its dependencies — model → storage → rules → identity → content → interaction → network → extras. No brick was ever laid before the brick it rests on.”
